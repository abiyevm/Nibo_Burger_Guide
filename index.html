Why this guide exists

Not another plain report — a real startup path.

The main problem for new NIBO students is not only writing code. It is knowing the correct order: install the right old environment, test each hardware part, calibrate sensors, then slowly move from line following to Micromouse. This guide is written like a student blog so the next group can avoid the same wasted time.

What confused us

Modern tools looked better, but they did not match the old NIBO Burger ecosystem. The board and library are made for Arduino 1.6x.

What solved it

Arduino IDE 1.6.7 / 1.6.x with complete NiboRoboLib installation, correct board, correct programmer, and simple test sketch.

What students should do

Start with LED and motor tasks, add sensor reading, build a stable line follower, then add junction logic and memory.

Chapter 01

Environment setup: the honest version.

The official tutorial is written for Arduino 1.6x. Our experience confirmed that this is not optional; it is the safest working path.

Attempt 1: VS Code + Arduino extension — failed.
We first tried Microsoft Visual Studio Code because it is modern and comfortable. However, the NIBO Burger custom board definitions and NIBOburger library were not resolved correctly. The extension was not reliable for this old robot ecosystem.
Attempt 2: latest Arduino IDE — failed.
The newer Arduino IDE 2.x can compile many modern Arduino projects, but the NiboRoboLib installation was designed for the older Arduino 1.x structure. The board/programmer options did not appear correctly.
Final working setup: Arduino IDE 1.6.x + NiboRoboLib complete installation.
This matched the official NIBO Burger tutorial and allowed us to select the board, programmer, include the library, verify code, and upload to the robot.

Exact setup steps for a new student

Install Arduino IDE 1.6.x

Use the Windows installer, not the ZIP version. Keep default options including USB drivers.

Install NiboRoboLib completely

Choose the complete installation so you get drivers, RoboDude, C libraries, Arduino libraries, examples, and board definitions.

Open Arduino IDE 1.6.x

Do not open Arduino 2.x by mistake if both are installed.

Select the board

Go to Tools → Board → nicai-systems NIBO burger robot.

Select the programmer

Go to Tools → Programmer → NIBObee & burger programmer (usbasp).

Include the library

Use Sketch → Include Library → NIBOburger, which adds #include <NIBOburger.h>.

Verify, then upload

First compile with the checkmark. Then turn on the robot, connect USB, and upload with the arrow button.

First upload test
#include <NIBOburger.h>

void setup() {
  NIBOburger.begin();          // must be first
  NIBOburger.checkVoltage();   // safety battery check
}

void loop() {
  NIBOburger.setLed(LED1, ON);
  delay(500);
  NIBOburger.setLed(LED1, OFF);
  delay(500);
}
Success sign: after upload, LED1 flashes on and off. This proves that the IDE, board, programmer, library, USB connection, and robot power are working.
Chapter 02

Programming foundation before line following.

Do not jump directly to PID or maze solving. First test every robot feature separately. These tasks build confidence and make debugging easier later.

A

Round Trip

Drive out, turn 180°, return. Teaches state machines and smooth speed ramping.

B

Motors

Use buttons to increase/reset/decrease motor PWM. Teaches global variables and limits.

C

Light Switch

Button-controlled LED patterns. Teaches input events and predictable output states.

D

Button Modes

Toggle flashing modes using millis(). Teaches non-blocking timing.

E

LED Animation

Travelling LED effect using arrays. Teaches sequences and helper functions.

F

Sensors

Read surface sensors and odometry. Teaches real robot feedback.

Recommended learning order
  1. First sketch: LED blink.
  2. LED patterns: learn outputs.
  3. Buttons: learn inputs.
  4. Motors: learn safe movement.
  5. Odometry: understand movement measurement.
  6. Surface sensors: understand line detection.
  7. Line follower: combine sensors + motors.
  8. Maze solver: combine line follower + decisions.
Non-blocking timing pattern
unsigned long lastStep = 0;
int interval = 300;

void loop() {
  unsigned long now = millis();
  if (now - lastStep >= interval) {
    lastStep = now;
    // do one small action here
  }
}
Why this matters: delay() freezes the whole robot. millis() lets the robot continue reading buttons and sensors while timing actions.
Chapter 03

Building a strong line follower.

The line follower is the central building block. A bad line follower means the maze solver will never be reliable.

1. Calibrate

Normalize left, center, and right surface sensor values so the robot works under current lighting and floor conditions.

2. Detect position

Classify the robot as CENTER, HARD_LEFT, HARD_RIGHT, RECOVERY_LEFT, RECOVERY_RIGHT, etc.

3. Correct motors

Use PID output to increase one motor and decrease the other smoothly.

Core PID idea

The robot computes the error with pid_input = rightSensor - leftSensor. If the right sensor sees more line than the left, the robot is off-center. PID transforms this error into a steering correction.

Line follower control formula
pid_input = r - l;
steerPID.Compute();

pwm_left  = constrain(BASE_SPEED + pid_output, MIN_PWM, MAX_PWM);
pwm_right = constrain(BASE_SPEED - pid_output, MIN_PWM, MAX_PWM);

Engine.setPWM(pwm_left, pwm_right);
StateMeaningRobot reaction
CENTEROnly center sensor sees the lineDrive forward with PID correction
CENTER_LEFTLeft and center sensors see lineCorrect gently to the right
CENTER_RIGHTCenter and right sensors see lineCorrect gently to the left
HARD_LEFTOnly left sensor sees lineSharp left correction
HARD_RIGHTOnly right sensor sees lineSharp right correction
RECOVERY_LEFT/RIGHTNo sensor sees lineSearch in last known direction
Best practical advice: keep LED feedback active during testing. When the robot lights the left LEDs, right LEDs, or outer LEDs, you can immediately see what the code thinks is happening.

Recommended tuning process

Start slow

Use a lower BASE_SPEED first. A slow robot is easier to debug.

Set KI = 0

Integral correction usually causes over-correction for beginner line following.

Increase KP

Raise it until the robot follows the line but starts to wiggle.

Add KD

Increase derivative slowly until the wiggle becomes stable.

Adjust thresholds

If the robot misses black lines, increase/decrease threshold depending on sensor values.

Chapter 04

From line follower to Micromouse.

Micromouse/maze solving is not a separate project. It is a line follower plus junction detection plus decision memory.

Level 1: Follow line

Stay on the path using PID and recovery states.

Level 2: Detect junctions

Recognize left, straight, right, and dead-end situations.

Level 3: Choose direction

Use left-hand priority first, then improve with memory/backtracking.

Simple decision rule

Left-hand priority
if (paths.left) {
  turnLeft();
} else if (paths.straight) {
  goStraight();
} else if (paths.right) {
  turnRight();
} else {
  turnAround(); // dead end
}

Advanced memory-based idea

For a better Micromouse-style solution, the robot should remember what it already tried at each junction. The memory version stores a usage count for each available arm: left, straight, right, and back. The robot chooses the least-used path instead of always turning left.

Junction memory structure
enum Arm : uint8_t {
  ARM_LEFT = 0,
  ARM_STRAIGHT = 1,
  ARM_RIGHT = 2,
  ARM_BACK = 3
};

struct JunctionRecord {
  uint8_t marks[4];   // how many times each direction was used
  bool valid;
};

JunctionRecord junctionMemory[32];
VersionBehaviorBest for
Left-hand ruleAlways prefers left, then straight, then right, then U-turnSimple demos and connected mazes
Memory solverTracks used arms and chooses less-used pathsSmarter exploration and dead-end handling
Optimized MicromouseMaps maze, then runs shortest pathCompetition-style future improvement
Upgrade idea for next students: after solving once, save the successful route and run the maze again using the shortest cleaned path. That would turn this from a simple maze solver into a real Micromouse-style project.
Chapter 05

Debugging guide: problems we expect.

Most NIBO problems are not mysterious. They usually come from setup, power, calibration, sensor thresholds, or turning time.

ProblemLikely reasonFix
Board missing in ArduinoNiboRoboLib not installed completelyReinstall complete NiboRoboLib
Upload fails / USBasp errorRobot not powered, driver issue, wrong programmerTurn on robot, select NIBObee & burger programmer, check USB
Code compiles but robot does nothingNIBOburger.begin() missing or not firstPut it first in setup()
Line follower loses the lineBad calibration or high speedRecalibrate, lower speed, tune thresholds
Robot spins wrong way after losing lineNo recovery memoryRemember last hard direction and search that way
Turns are not 90 degreesTURN_90_MS not tuned for floor/frictionAdjust by 20 ms steps
Sensors act strangeLow battery or lighting changedCheck voltage and recalibrate

Final run checklist

Correct IDE open?

Arduino 1.6.x, not Arduino 2.x.

Correct board and programmer?

NIBO burger robot board and NIBObee & burger programmer selected.

Battery okay?

Call NIBOburger.checkVoltage() and avoid testing with low battery.

Sensor calibration done?

Calibrate every time floor, light, or sensor position changes.

Start with slow speed?

Increase speed only after the robot is stable.
